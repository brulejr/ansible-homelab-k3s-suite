# ansible-homelab-k3s-suite

An Ansible-driven homelab deployment suite for running services on a K3s-based Kubernetes platform.

## Current status

This repository is currently operating in **Ansible-only apply mode**.

That means:

- Ansible installs and configures K3s
- Ansible installs Traefik via Helm
- Ansible applies the Kubernetes overlay directly with `k3s kubectl apply -k`
- Flux/GitOps scaffolding exists in the repository but is **not currently part of the active deployment path**

## Current working baseline

The current working baseline is the `minidev_servers` site with the `minidev-attached` overlay.

Working components include:

- K3s single-node cluster
- Traefik ingress controller
- PlantUML
- Mosquitto (MQTT)
- Wiki.js
- PostgreSQL for Wiki.js

## Usage

Run the following to deploy this playbook:

```bash
ansible-playbook --ask-vault-pass -i inventories/examples/minidev/hosts.yml site.yml
```

## Deployment flow

The active deployment flow is:

1. `storage_prep`
2. `k3s_server`
3. `traefik`
4. `k8s_apply`

The Kubernetes manifests are applied from:

- `kubernetes/overlays/minidev-attached`
- or another overlay selected via `k8s_overlay_name`

## Secrets strategy

This repository currently uses two secret-handling patterns.

### 1. Vaulted inventory secrets

Cluster/bootstrap secrets are stored as vaulted inventory values.

Examples:

- `k3s_server_token`

### 2. Generated app credentials

Some app-local credentials are generated by Ansible on the control machine and persisted under:

- `.secrets/<overlay-name>/`

These values are gitignored and rendered into staged Kubernetes Secret manifests at deploy time.

Example:

- Wiki.js / PostgreSQL database password

## Overlays

Current overlays:

- `minidev-attached`
- `minidev-portable`

These overlays represent different deployment modes for the same mini rack, not simultaneous side-by-side environments.

## Rendered-at-deploy files

Some files in the committed Kubernetes tree are placeholders only and are overwritten during Ansible staging on the target host.

Currently rendered by `roles/k8s_apply`:

- overlay secret manifests for Wiki.js/PostgreSQL
- route manifests for PlantUML and Wiki.js hostnames

The committed versions exist to preserve the kustomize tree structure, but inventory values and controller-local generated secrets are the real source of truth at deploy time.

## Example inventories

The inventories under `inventories/examples/` are sample-only and are intended to show structure and expected variables.

They are not meant to be the long-term private operational inventory for a real deployment.

In particular:

- example vault files are provided as `vault.yml.example`
- real vaulted secrets should live in a private inventory repository
- controller-local generated app secrets remain under `.secrets/` and are gitignored

## Future direction

The planned future direction is:

- split private site-specific inventory into a separate repository
- keep this repository as the reusable suite
- reintroduce Flux/GitOps once the suite and inventory are separated cleanly
